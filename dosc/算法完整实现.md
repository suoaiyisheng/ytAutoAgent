完整的算法过程：
第一阶段：结构化提取 (Structural Extraction)

- 目标：将非结构化视频转化为可处理的物理片段。
- 输入：视频 URL / 原始 MP4。
- 算法：yt-dlp (下载) -> PySceneDetect (内容感应检测) -> FFmpeg (精准抽帧)。
- 输出【契约 1：物理清单 Physical Manifest】：
  - video_id, fps, resolution。
  - scenes: [{scene_id, start_time, end_time, keyframe_path, video_clip_path}]。

---

第二阶段：原子级感知 (Atomic Perception)

- 目标：利用多模态大模型（VLM）“看图说话”，提取原始语义。
- 输入：Physical Manifest 中的图片。
- 算法：调用 Gemini 1.5 Pro 进行 Batch 分析。
- 输出【契约 2：原始描述 Raw Scene Description】：
  - 每个分镜的原子描述：subjects (外观特征列表)、environment、camera_info。
  - 注：此时角色 ID 还是随机的（如 subject_A），不具备全局一致性。

---

第三阶段：身份对齐与归一化 (Identity Alignment & Normalization)

- 目标：【核心补丁】 将散乱的描述聚类，建立全局“身份证”。
- 算法：
  1. 角色提取：提取所有分镜中出现的人物特征。
  2. 语义聚类：利用 LLM 或 Embedding 算法判断：“穿红领带的黄色方块” == “海绵宝宝”。
  3. 资产库建立：确定 Ref_1, Ref_2 的标准定义。
- 输出【契约 3：全局资产库 Character Bank】：
  - characters: [{ref_id, global_description, best_face_image}]。
- 输出【契约 4：对齐分镜脚本 Aligned Storyboard】：
  - 将 Raw Scene Description 中的 subject_A 替换为确定的 Ref_1。

---

第四阶段：指令合成 (Prompt Synthesis)

- 目标：调用你的 V4.2 架构师提示词 输出最终结果。
- 输入：Character Bank + Aligned Storyboard。
- 算法：模板渲染 + Gemini 系统指令。
- 输出【契约 5：最终生成表格 Final Production Table】：
  - Image Prompt (对应起始帧) 和 Video Prompt (对应动态)。
    这五份 JSON 契约共同构成了你自动化视频流的“单一事实来源（Single Source of Truth）”。作为开发者，你可以将它们视为一套协议栈，每一层都为下一层消除不确定性。
    契约

1. 物理清单契约 (Physical Manifest)
   作用： 解决文件定位问题。它是流水线的“硬件层”输出。
   JSON
   {
   "project_id": "string (uuid)",
   "video_metadata": {
   "source_url": "string",
   "fps": "number",
   "resolution": "string (e.g. 1920x1080)"
   },
   "scenes": [
   {
   "scene_id": "number",
   "start_time": "string (HH:MM:SS.mmm)",
   "end_time": "string (HH:MM:SS.mmm)",
   "keyframe_path": "string (local_path)",
   "clip_path": "string (local_path)"
   }
   ]
   }

---

2. 原始描述契约 (Raw Scene Description)
   作用： 将视觉信号转为原子文字描述。它不负责“认人”，只负责“客观描述”。
   JSON
   {
   "scene_id": "number",
   "visual_analysis": {
   "subjects": [
   {
   "temp_id": "string (e.g. subject_A)",
   "appearance": "string (详细的外貌、服饰特征描述)",
   "action": "string (当前动作趋势)",
   "expression": "string (起始表情描述)"
   }
   ],
   "environment": {
   "location": "string",
   "lighting": "string",
   "atmosphere": "string"
   },
   "camera": {
   "shot_size": "string",
   "angle": "string",
   "movement": "string"
   }
   }
   }

---

3. 全局资产库契约 (Character Bank)
   作用： 【对齐核心】 存储归一化后的角色定义。它决定了 AI 在生成时对“某个人”的固有印象。
   JSON
   {
   "characters": [
   {
   "ref_id": "string (e.g. Ref_1)",
   "name": "string (optional)",
   "master_description": "string (融合多镜头的最终标准视觉描述)",
   "key_features": ["string", "string"], // 必须保持一致的视觉锚点"ref_image_path": "string (该角色的最佳参考图路径)"
   }
   ],
   "global_style": "string (如：电影感写实, 皮克斯3D)"
   }

---

4. 对齐分镜脚本契约 (Aligned Storyboard)
   作用： 衔接层。将“原始动作”与“全局 Ref_ID”挂钩。这是喂给 Gemini-Sora 架构师 V4.2 的直接输入。
   JSON
   {
   "project_id": "string",
   "storyboard": [
   {
   "shot_id": "number",
   "character_mappings": [
   {
   "ref_id": "string (Ref_1)",
   "action_in_shot": "string",
   "expression_range": ["start_expression", "end_expression"]
   }
   ],
   "environment_context": "string",
   "camera_instruction": "string"
   }
   ]
   }

---

5. 最终生成表格契约 (Final Production Table)
   作用： 最终产物。符合你提示词模板要求的、可直接用于视频生成的指令。
   JSON
   {
   "prompts": [
   {
   "shot_id": "number",
   "image_prompt": "string (遵循：序号，[风格], [景别], [视角]。**[角色代号]**...)",
   "video_prompt": "string (遵循：[运镜方式]，[主体代词][动作1], [动作2]...)"
   }
   ]
   }
